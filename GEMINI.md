# 週次日報＆予定管理システム (weekly-report-app)

## 1. プロジェクト概要
本プロジェクトは、図面作図業務における各担当者の週次作業日報（実績）および月間作業予定（スケジュール）を効率的に登録・管理・集計するためのWebアプリケーションです。

### 主な機能
1. **認証機能 (Firebase Auth)**: 新規登録時の「フルネーム（氏名）」登録とプロフィール（DisplayName）紐付けに対応。メンバー自身によるアカウントの新規登録（方法B）およびログイン・ログアウト機能。
2. **予定登録＆ガントチャート**: 担当者ごとの工事予定を入力し、月間の作業予定をガントチャート（A3横印刷対応）で俯瞰する機能。日付入力用の便利なショートカットボタン（今日、明日、来週月曜）付き。
3. **実績入力（週次日報）**: 曜日ごとの稼働工事名・作業内容・稼働時間に加え、各曜日のカード内にその日の作業報告・備考を直接入力し、提出する機能（従来の画面最下部の週全体のまとめ欄は廃止）。
   - **工事名サジェスト**: 過去のデータから工事名を自動補完。
   - **日報コピー機能**: 過去に提出した日報から、作業内容と日次レポート内容をワンクリックで複製して入力時間を短縮。
   - **リアルタイム時間計算**: 各日の合計稼働時間、およびヘッダー部分の「今週の合計稼働時間」を即座に自動計算。
4. **個人別一覧・レポート（月間集計）**: 月単位での日報の確認、A4縦での印刷用レイアウト、および担当者別の月間工事稼働時間（合計）の自動集計表示。
5. **工事別集計**: 工事ごとに、どの週に誰が何時間作業したかを一元的に確認できる集計テーブル。
6. **Excelエクスポート**: 各一覧・集計データをボタン一つでExcelファイル（複数シート構成含む）に出力（SheetJSを使用）。
7. **マルチデバイス対応（iPad最適化）**: iPadやスマートフォンなどタッチ操作の多い端末でも崩れず使いやすいボタン・入力欄レイアウト。
8. **ダークモード**: 端末や好みに合わせてライト・ダークテーマを切り替える機能（設定はブラウザに自動保存）。

## 2. 使用技術
本システムはサーバーレスなSPA（シングルページアプリケーション）として構築されています。

- **コア言語**: HTML5, Vanilla CSS, Vanilla JavaScript (ES Modules)
- **バックエンド / データベース**: Firebase (v10.8.0)
  - **Firebase Authentication**: ユーザー認証管理
  - **Cloud Firestore**: スケジュールおよび日報データの永続化
- **ライブラリ**:
  - **SheetJS (xlsx.full.min.js)**: クライアントサイドでの高度なExcelファイル生成
- **インフラ・公開環境**:
  - **GitHub**: ソースコード管理 (`https://github.com/1105yuky00109-web/weekly-report-app`)
  - **Netlify**: ホスティング＆継続的デプロイ (CI/CD)

## 3. ビルド・デプロイルールと構成
セキュリティの観点から、APIキーなどのFirebase接続情報はソースコードに直接コミットせず、環境変数からビルド時に動的注入する仕組みを採用しています。

### ディレクトリ構成
```text
├── public/                 # 公開ディレクトリ (Netlify公開対象)
│   ├── index.html          # アプリケーションのメインHTML
│   ├── app.js              # フロントエンドロジック (ビルド時に接続キーが注入される)
│   └── style.css           # スタイルシート (ダークモード・iPad・印刷用定義含む)
├── build-inject.js         # ビルド時環境変数注入用Node.jsスクリプト
├── package.json            # プロジェクト設定 (ビルドスクリプト定義)
├── netlify.toml            # Netlify用ビルド・デプロイ設定
└── .gitignore              # Git管理対象外ファイル定義
```

### ビルドの仕組み
Netlifyでビルドが行われる際、`package.json` に定義された `npm run build` (内部的に `node build-inject.js` を実行) が走ります。
このスクリプトは、Netlify側に設定された環境変数から以下のFirebase設定を読み取り、`public/app.js` のプレースホルダー部分に注入します。

#### 設定が必要な環境変数 (Netlifyダッシュボード上)
- `FIREBASE_API_KEY`
- `FIREBASE_AUTH_DOMAIN`
- `FIREBASE_PROJECT_ID`
- `FIREBASE_STORAGE_BUCKET`
- `FIREBASE_MESSAGING_SENDER_ID`
- `FIREBASE_APP_ID`

## 4. 開発時のルール・規約
コードを編集・改修する際は以下のルールを厳守してください。

1. **言語設定**:
   - ソースコード内のコメントやDocstringは、原則として**日本語**で記述してください。
   - ユーザーとの対話、ドキュメントもすべて日本語を使用します。
2. **依存ライブラリの追加**:
   - 余計な依存パッケージ（Node.jsモジュール等）は極力追加せず、HTMLから直接CDNで読み込む軽量な構成を維持してください。
3. **レスポンシブデザインの維持**:
   - iPad（768px〜1024px）およびモバイル（600px以下）のレイアウトが崩れないよう、CSS編集時はメディアクエリの動作を確認してください。
   - タッチ操作時に押しづらくならないよう、クリックターゲット（ボタンや選択肢）は `40px` 以上の高さを確保してください。
4. **セキュリティルール**:
   - Firestoreは認証されたユーザーのみが読み書きできるように、Firebase側で適切なセキュリティルールを維持してください。
5. **Excel出力（ExcelJS）時のスタイル共有バグの防止**:
   - 既存のExcelテンプレート（`template-weekly.xlsx`など）を読み込んでセルのスタイル（塗りつぶしなど）を上書きする際、Excelの最適化仕様により、複数のセルが裏で「同じスタイルオブジェクト」を共有していることがあります。
   - スタイルを共有したまま1つのセルの `fill` や書式を変更すると、他の共有セルも連動して書き換わってしまいます。
   - これを防ぐため、ループ処理などでセルにスタイルを設定する際は、必ず事前に `cell.style = Object.create(cell.style);` を実行して、スタイル参照の共有を解除（デカップリング）してください。
